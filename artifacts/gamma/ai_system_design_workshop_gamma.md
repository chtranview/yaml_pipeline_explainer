# 你現在做的很多事，其實都已經是「系統」

# 核心概念 1：模組化設計

## 把複雜流程拆成可替換的小單元

### 不是：
「我整個做完」

### 而是：
- 需求整理
- 架構生成
- 草稿輸出
- 檢查修正
- 最終交付

### 關鍵問題
> 哪些步驟可以被獨立處理？

---

# 核心概念 2：I/O 邊界

## 每個模組都要說清楚：
### 它吃什麼？吐什麼？

### 範例
**模組：課程大綱生成**
- Input：課程主題、受眾、天數
- Output：課程 outline.yaml

### 如果 I/O 不清楚
整個流程就會開始亂掉

---

# 核心概念 3：依賴關係

## 不是所有模組都能同時做

### 例子
- 沒有 outline → 無法產生 handout
- 沒有 Skill → 很難寫 YAML
- 沒有 MVP 範圍 → Demo 會失焦

### 系統設計常見錯誤
> 還沒定義前面，就急著做後面

---

# 核心概念 4：流程抽象化

## 把「我平常怎麼做」
轉成
## 「別人也能照著做」

### 這一步很重要
因為 AI 最適合接手的，不是靈感  
而是 **可描述的工作流程**

# Day 1｜模組 2
# 從 Prompt 到 Skill：設計可重用的 AI 任務單元

---

# 為什麼很多 Prompt 不穩？

## 因為它只是「一次性交代」
不是「可重複執行的規格」

### Prompt 常見問題
- 太依賴當下情境
- 沒有固定輸出格式
- 換人就失效
- 很難交接或複製

---

# Prompt vs Skill

## Prompt
### 一次性任務交代

例如：
> 幫我整理這篇文章

## Skill
### 可重複執行的任務規格

例如：
> 當我貼入文章時，請輸出摘要、重點、行動建議，格式固定

---

# 一個 Skill 至少要有 4 件事

## 1. Trigger
什麼情況下啟動？

## 2. Instruction
AI 應該怎麼處理？

## 3. Input
需要什麼資料？

## 4. Output Contract
最後要長什麼樣？

---

# Trigger：什麼時候該啟動這個 Skill？

## 沒有 Trigger，Skill 就會很鬆散

### 範例
- 當我輸入 `/summary` 並貼上一段文章
- 當我上傳會議逐字稿
- 當我給你課程主題與受眾

### Trigger 的價值
> 幫你界定這個 Skill 的使用場景

---

# Instruction：不要期待 AI 自己猜懂

## 好的 Instruction 不是更長
而是更清楚

### 常見寫法
- 先辨識主題
- 再整理重點
- 最後產出行動建議

### 關鍵
> 要寫成「可以被執行」的步驟

---

# Output Contract：先定義完成長什麼樣

## 不先定義輸出，品質就很難穩定

### 範例
請固定輸出：
- 一句話摘要
- 三個重點
- 一個下一步建議

### 重點
> AI 不怕你要求多，AI 怕你要求不明確

---

# Skill 範例頁

## 範例：會議摘要助手

- **Skill Name**：Meeting Summary Assistant
- **Trigger**：當使用者貼上逐字稿時
- **Input**：meeting_transcript
- **Instruction**：
  - 辨識主題
  - 整理決策
  - 列出待辦事項
- **Output**：
  - Summary
  - Key Decisions
  - Action Items

# Day 2｜模組 3
# YAML 作為 AI 系統藍圖：把流程變成可執行規格

---

# 為什麼要學 YAML？

## 因為你需要的不只是「提示詞」
而是 **系統規格**

### YAML 的價值
- 結構清楚
- 可維護
- 可交接
- 可擴充
- 適合被系統或團隊重用

---

# 從 Skill 到 YAML

## Skill 是任務設計
## YAML 是系統藍圖

### Skill 回答的是：
> 這件事要怎麼做？

### YAML 回答的是：
> 這個模組在系統裡怎麼運作？

---

# 一個最小可行 YAML 模組至少要有：

- `module_id`
- `description`
- `inputs`
- `steps`
- `outputs`
- `validation`
- `error_handling`

### 這不是格式練習
### 這是在練「結構化思考」

---

# 範例頁：YAML 模組骨架

```yaml id="ztxhh0"
module_id: meeting_summary_generator
version: "1.0"
description: >
  將會議逐字稿轉成結構化摘要

inputs:
  - field: transcript
    type: string
    required: true

steps:
  - id: identify_topic
    description: "辨識會議主題"
  - id: summarize_key_points
    description: "整理關鍵決策"
  - id: extract_actions
    description: "列出待辦事項"

outputs:
  summary:
    format: markdown

validation:
  - "摘要不得超過 300 字"

error_handling:
  on_missing_transcript: "要求使用者補上逐字稿"
