0_build_pipeline.yaml
一鍵建置流程 · 總指揮
▼
YAML 結構
trigger: "/build" description: "主控入口，接收指令 後依序呼叫所有子模組" sequence: - call: 1_project_engine - call: 2_learner_handout_generator - call: 3_sdd_auto_generator - call: 4_gamma_generator - call: 6_verifier_auto_checker - call: 7_packager_auto_zipper - call: 5_notifier_auto_dispatch on_error: abort_and_log
職責與運作原理
  • 🎯使用者輸入 /build 後，此模組作為「總排程器」依序觸發所有子流程
  • 🔗每個 call 都是同步呼叫：前一步完成才進入下一步
  • 🛑on_error: abort_and_log — 任一步驟失敗則中止並記錄錯誤至 _logs/
  • 💡在 ChatGPT 中，這個 YAML 被貼入 System Prompt，GPT 解讀後模擬「執行」各步驟
INPUT/build + 課程主題
OUTPUT觸發完整 pipeline 執行

1_project_engine_4.0.yaml
課程骨架 × 教案引擎
▼
YAML 結構
module_id: project_engine version: "4.0" inputs: - field: course_topic required: true - field: duration_days default: 2 - field: audience_level options: [beginner, intermediate, advanced] steps: - define_learning_objectives - break_into_modules # 每模組獨立 I/O - sequence_by_dependency - assign_time_blocks output: path: artifacts/handouts/outline.yaml also_passes_to: [2, 3, 4]
職責與運作原理
  • 🏗️整個系統的「地基」——先定義課程結構，後續模組才知道要生成什麼
  • 📐將課程拆解為獨立模組，每個模組有明確的學習目標、時長、先後依賴
  • 📤also_passes_to: [2, 3, 4] — 輸出的 outline.yaml 同時被後三個模組當作輸入
  • 🔢version 4.0 表示這是第四代迭代，欄位設計趨於穩定
INPUTcourse_topic, duration_days, audience_level
OUTPUTartifacts/handouts/outline.yaml
FEEDS模組 2、3、4 同時讀取此輸出

2_learner_handout_generator.yaml
學員講義生成器
▼
YAML 結構
depends_on: 1_project_engine reads: artifacts/handouts/outline.yaml template: prompts/handout_template.md loop_over: outline.modules # 每模組跑一次 per_module_output: format: [yaml, md] includes: - objectives # 學習目標 - key_concepts # 核心概念 - hands_on_task # 動手實作 - takeaway # 帶走成果 - reflection_qs # 反思問題 output_path: artifacts/handouts/{module_id}.md
職責與運作原理
  • 📄loop_over: outline.modules — 對每個模組「迴圈」執行，自動批量生成所有講義
  • 🗂️每份講義包含五個固定欄位，確保格式一致、品質可驗證
  • 📁以 {module_id} 命名輸出檔，避免衝突且易於追蹤
  • 🔄同時輸出 YAML（機器讀）+ MD（人讀），供後續模組引用
DEPENDSoutline.yaml（來自模組 1）
OUTPUTartifacts/handouts/{module_id}.md
3
3_sdd_auto_generator.yaml
App SDD 自動化生成
▼
YAML 結構
purpose: "將課程實作任務轉化為 可執行的 App 規格文件" reads: - artifacts/handouts/outline.yaml - artifacts/handouts/*.md # 所有講義 extract: hands_on_task # 只取實作欄位 sdd_schema: sections: - app_name_and_goal - user_stories - feature_list - data_model - ui_wireframe_description - success_criteria output_path: artifacts/sdd/{module_id}_sdd.yaml
職責與運作原理
  • 📋SDD = Software Design Document，把「課程實作任務」升格為可開發的 App 規格
  • 🔍extract: hands_on_task — 只擷取講義中的實作欄位，不重複處理理論內容
  • 🏛️固定的六段式 schema 讓每份 SDD 結構統一，學員可直接拿去開發
  • ✅最後一欄 success_criteria 讓學員知道「完成」長什麼樣
DEPENDSoutline.yaml + 所有 handout.md
OUTPUTartifacts/sdd/{module_id}_sdd.yaml
4
4_gamma_generator.yaml
投影片文字生成器（Gamma 用）
▼
YAML 結構
target_tool: "Gamma.app" reads: artifacts/handouts/*.md slide_rules: words_per_slide: max 40 structure: - title_slide - agenda - concept_slides # 1 概念 = 1 頁 - hands_on_slide # 實作說明頁 - takeaway_slide tone: "direct, visual-first, no filler" language: zh-TW output_path: artifacts/gamma/{module_id}_deck.md
職責與運作原理
  • 🖼️專門為 Gamma.app 生成適合的投影片文本，非通用文字輸出
  • 📏words_per_slide: max 40 — 硬性限制每頁字數，確保視覺簡潔
  • 🔁從講義（長文）自動壓縮成簡報（短文），是「格式轉換」而非重新生成
  • 🌐明確指定 zh-TW，避免混用繁簡或英文
DEPENDSartifacts/handouts/*.md
OUTPUTartifacts/gamma/{module_id}_deck.md

5_notifier_auto_dispatch.yaml
學員通知 × Email / LINE 自動寄送
▼
YAML 結構
trigger: "after 7_packager completes" channels: - type: email template: prompts/email_notify.md to: "{{learner_list}}" - type: line_message template: prompts/line_notify.md webhook: "{{LINE_WEBHOOK_URL}}" message_includes: - course_name - bundle_download_link # 來自 packager - schedule_reminder fallback: log_to_artifacts/_logs
職責與運作原理
  • 📨pipeline 最末端的「對外介面」——產出物完成後自動通知相關人員
  • 🔗在 ChatGPT 中：生成通知文字草稿；實際寄送需搭配 Zapier / Make.com webhook
  • 📦帶入 packager 輸出的下載連結，讓學員直接取得材料
  • 🛡️fallback: log — 寄送失敗不中斷流程，改為記錄錯誤
DEPENDS7_packager 完成後觸發
OUTPUTEmail + LINE 通知 / 記錄至 _logs/

6_verifier_auto_checker.yaml
產物品質檢核與修復
▼
YAML 結構
checks: - rule: all_modules_have_handout source: artifacts/handouts/ on_fail: re_run module 2 - rule: no_missing_fields required_fields: - objectives - hands_on_task - takeaway on_fail: flag_to _issues/ - rule: yaml_schema_valid on_fail: auto_fix_or_escalate report_path: artifacts/_reports/verify.md
職責與運作原理
  • 🔍pipeline 的「品管站」——確認所有產物符合規格才允許打包
  • ♻️on_fail: re_run module 2 — 驗證失敗可自動重跑上游模組（自愈機制）
  • 📋生成 verify report，讓人類審閱哪些項目通過、哪些需人工確認
  • 🗂️問題記錄到 _issues/，形成可追蹤的錯誤清單
INPUTartifacts/ 下所有產物
OUTPUT_reports/verify.md + _issues/

7_packager_auto_zipper.yaml
模組成果打包 ZIP 交付
▼
YAML 結構
depends_on: 6_verifier # 驗證通過才打包 include: - artifacts/handouts/** - artifacts/sdd/** - artifacts/gamma/** - artifacts/_reports/verify.md exclude: - artifacts/_logs/** # 內部紀錄不交付 - artifacts/_issues/** # 錯誤清單不交付 - artifacts/_queues/** bundle_name: "{{course_topic}}_{{date}}.zip" output_path: artifacts/bundles/
職責與運作原理
  • 📦只打包「交付給學員」的檔案，內部日誌、錯誤記錄一律排除
  • 🔐depends_on: 6_verifier — 品管未通過則此步驟不執行（防呆設計）
  • 📅以日期命名 bundle，方便版本管理與重新交付
  • ➡️完成後觸發 5_notifier 寄出下載連結
DEPENDS6_verifier 通過
OUTPUTartifacts/bundles/課程名_日期.zip

8_bridge_connector.yaml
外部資料橋接（Sheets / Airtable / Notion）
▼
YAML 結構
connections: - name: google_sheets type: read source: "{{SHEET_ID}}/learner_roster" maps_to: learner_list - name: airtable type: write target: "{{AIRTABLE_BASE}}/course_log" writes: pipeline_run_summary - name: notion type: write target: "{{NOTION_PAGE_ID}}" writes: artifacts/_reports/verify.md auth: env_vars # 不硬寫在 YAML 裡
職責與運作原理
  • 🌉「橋接層」——讓 AI 系統與外部真實工具雙向溝通
  • 📥從 Google Sheets 讀取學員名單，注入到 notifier 的 learner_list
  • 📤執行摘要寫回 Airtable / Notion，形成可追蹤的課程歷程
  • 🔑auth: env_vars — 憑證不寫死在 YAML，透過環境變數注入（安全設計）
INPUTGoogle Sheets 學員名單（外部）
OUTPUT寫入 Airtable / Notion（外部）

9_meta_loop.yaml
課程更新 × 持續學習追蹤
▼
YAML 結構
purpose: "讓系統從每次執行中學習， 自動改善下一輪品質" reads: - artifacts/_reports/verify.md - artifacts/_logs/** - artifacts/_issues/** analyzes: - recurring_issues # 反覆出現的錯誤 - slow_steps # 耗時最長的步驟 - missing_fields_rate # 欄位缺失率 outputs: - path: artifacts/_queues/improvement.yaml content: "下一版本建議修改清單" - path: artifacts/_reports/meta_summary.md loops_back_to: 1_project_engine # 形成閉環
職責與運作原理
  • 🔁整個 pipeline 的「反饋迴路」——讀取所有日誌與錯誤，生成改善建議
  • 🧠分析反覆出現的問題 → 自動更新 _queues/ 中的待辦改善清單
  • ♾️loops_back_to: 1_project_engine — 建議回饋給引擎，讓下一次執行品質更高
  • 📈這是系統從「工具」進化到「自我優化引擎」的關鍵模組
INPUT_reports/ + _logs/ + _issues/
OUTPUT_queues/improvement.yaml
LOOPS TO1_project_engine（下一輪）

03 ChatGPT 如何「執行」這套 YAML 系統

YAML 本身不是程式，ChatGPT 扮演「解讀者 + 執行者」角色

01
將 YAML 貼入 System Prompt
把 _index.yaml 或各模組 YAML 貼入 ChatGPT 的系統提示詞，讓 GPT 「閱讀並記住」整套規則
System: 你是 Expert Course Engine。 以下是你的執行規範，請嚴格遵守： [貼入 YAML 內容]
02
使用者下達觸發指令
輸入 /build 加上課程主題，GPT 解讀為「執行 0_build_pipeline，帶入以下參數」
User: /build 課程主題：AI 系統設計工作坊 天數：2 受眾：中階
03
GPT 模擬流水線執行
GPT 依照 pipeline 的 sequence 順序，逐步「假裝」執行各模組，輸出對應格式的內容
▶ 執行 1_project_engine... ✓ outline.yaml 已生成 ▶ 執行 2_handout_generator... ✓ M1_intro.md 已生成
04
人工複製輸出 → 真實檔案
GPT 的輸出是「文字模擬」，需要手動複製到本地資料夾；或搭配 MCP / API 自動寫檔
# 進階：搭配 MCP 自動寫檔 # Claude Desktop → MCP Filesystem # → 直接寫入 /artifacts/
05
驗證 → 打包 → 通知
繼續輸入 /verify、/pack，GPT 模擬 6、7 模組執行品質檢核與打包清單
User: /verify ▶ 執行 6_verifier... ✓ 所有模組含 takeaway ⚠ M3 缺少 hands_on_task → 記錄至 _issues/
06
Meta Loop 閉環優化
每次執行後輸入 /meta，GPT 分析本次問題並生成下一版本的改善建議，寫入 _queues/
User: /meta ▶ 分析 _issues/ + _logs... 建議：M3 模板加入 hands_on_task 必填驗證 → improvement.yaml 已更新
04 實戰 Prompt 範例（可直接使用）

完整的 ChatGPT System Prompt 架構，搭配課程內容直接貼入使用

ChatGPT System Prompt
# 角色定義 你是 Expert Course Engine v4.0，一個專門生成 AI 課程材料的自動化系統。 你的行為由以下 YAML 規範嚴格控制。 # 觸發指令對照表 "/build" → 執行完整 pipeline（模組 1 → 7） "/outline" → 只執行模組 1（生成課程骨架） "/handout" → 只執行模組 2（生成指定模組講義） "/sdd" → 只執行模組 3（生成 App 規格） "/deck" → 只執行模組 4（生成投影片文本） "/verify" → 執行模組 6（品質驗證） "/pack" → 執行模組 7（列出打包清單） "/meta" → 執行模組 9（分析並生成改善建議） # 輸出格式規則 - 每個模組輸出前，先打印：▶ 執行 {module_id}... - 完成後打印：✓ {output_filename} 已生成 - 遇到缺失欄位：⚠ {field} 缺失 → 記錄至 _issues/ - 輸出內容以 yaml 或 markdown 代碼塊包覆 # 執行約束 - 必填欄位缺少時，停下來詢問，不要自行填充假資料 - 每個模組輸出後等待確認，再進入下一模組 - 不在輸出中加入任何解釋或閒聊，只輸出規定的結構 # 記憶範圍 本次對話中所有已生成的 artifacts 視為「已存在於系統中」， 後續模組可直接引用前面生成的內容。
